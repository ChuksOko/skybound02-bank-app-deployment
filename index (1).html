02 — Architecture

System Architecture

Three containerised components deployed on AWS EKS across dedicated Kubernetes namespaces, following GitOps principles.

User
Hits https://bank.skybound02.online or https://bankapi.skybound02.online
Route 53
Resolves domain to AWS Network Load Balancer (NLB)
NLB
Forwards traffic to NGINX Ingress Controller inside EKS cluster
NGINX Ingress
/ → Frontend (frontendapp namespace)  ·  /api → Backend (backendapp namespace)
ArgoCD
Watches Repo C · Syncs manifests to EKS · Dashboard at argocd.skybound02.online

# frontend.yaml — CI workflow
name: Build and Push Frontend
on:
push:
branches: [ main ]
jobs:
build:
runs-on: ubuntu-latest
steps:
- name: Checkout code
uses: actions/checkout@v2
- name: Build Docker image
run: docker build -t ghcr.io/org/frontend:latest .
- name: Login to GHCR
run: echo "${{ secrets.GHCR_PAT }}" | docker login ghcr.io -u USERNAME --password-stdin
- name: Push to GHCR
run: docker push ghcr.io/org/frontend:latest

05 — Challenges & Resolutions

Bugs We Solved

01
Terraform S3 Backend Error
Error: No valid credential sources found for AWS Provider
Cause
Terraform could not find valid AWS credentials during terraform init — credentials not configured in the GitHub Actions runner environment.
Resolution
Updated the Terraform backend configuration with correct S3 bucket, key, region, and DynamoDB table. Added AWS credentials as GitHub Actions secrets.
02
Terraform Validation Error — helm_release
Misunderstood syntax: helm_release does not support set {} block
Cause
The helm_release resource expected a key-value pair, not a nested block. Terraform validated against the wrong structure.
Resolution
Corrected addons.tf to use set = [{ name = "installCRDs" value = "true" }] as a list assignment instead of a nested block.
03
ECR Authentication Error
aws: command not found · Cannot perform an interactive login from a non TTY device
Cause
AWS CLI was not installed on the GitHub Actions runner. Docker login failed because GitHub Actions runs in a non-interactive (non-TTY) environment.
Resolution
Added an explicit AWS CLI installation step in the workflow and used get-login-password for non-interactive Docker authentication.
04
Kubernetes Manifest Checkout Error
Pipeline failed at checkout stage when applying frontend.yaml and backendapi.yaml
Cause
Incorrect or missing Amazon ECR image references in the Kubernetes manifests — the image paths did not match the actual ECR account structure.
Resolution
Updated image references in both frontend.yaml and backendapi.yaml to point to the correct ECR image path under the account.
05
MySQL Database Connection Error
Backend could not connect to MySQL — application startup failed
Cause
Incorrect spring.datasource.url in application.properties — wrong hostname, port, or database name for the running MySQL instance.
Resolution
Updated the database connection URL in application.properties to correctly point to the running MySQL pod with the right host, port, and database name.
06 — Lessons Learned

Key Takeaways

01
Real-World Debugging Skills
Resolving errors like AWS CLI authentication, Terraform validation failures, and GitHub Actions deprecation taught us to read logs carefully and trace problems back to their root cause.
02
Value of Infrastructure as Code
Using Terraform reinforced how automation reduces manual configuration errors. Writing reusable, modular code improved consistency across the deployment.
03
CI/CD Pipeline Design
Configuring multi-stage pipelines taught us how to pass secrets securely, avoid interactive commands in non-TTY environments, and handle deprecation warnings proactively.
04
Cross-Functional Collaboration
Coordinating across Architecture, AWS, and DevOps sub-teams showed how clear documentation prevents duplicated effort and supports team accountability during sprint cycles.
05
Cloud Provider Specifics
Configuring S3 backends, managing state in DynamoDB, and authenticating with ECR demonstrated how cloud integrations require deliberate, careful setup — small details matter.
06
Iterative Debugging
Not every solution worked first time. Small, testable changes and using logs to guide fixes proved more effective than making large changes and hoping for the best.
